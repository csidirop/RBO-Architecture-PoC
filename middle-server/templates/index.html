Image Access PoC

Apache on the middle server authenticates the user with Keycloak, this PHP app applies the local policy, and the Apache file server validates a short-lived bearer token before it serves protected content.

user must have the role . File token lifetime: seconds.

Demo Accounts

  • reader / reader123 can request protected files.
  • blocked / blocked123 can log in but will be denied by policy.
  • admin / admin opens the Keycloak admin console at port 8081.

These demo users are auto-created from the Keycloak realm import when the PoC starts with a fresh Keycloak container.

Flow

  1. User signs in with Keycloak.
  2. Apache forwards authenticated identity claims to the middle service.
  3. Middle service checks the Keycloak role.
  4. Middle service mints a short-lived file token for one file.
  5. Apache on the file server validates that token and serves the file.

Protected Files

Links for quick testing. Browser clicks stay on the middle-service route, which streams the file back after a header-based handoff to the file server. If no session exists yet, the link falls back to the normal login redirect.

Browser navigation never needs the token in the URL. API clients can still call /protected/<file> directly with Authorization: Bearer <token>.
